# Adding a New Job Source Plugin

## Overview

Job sources are self-contained plugins discovered automatically at startup. Each
plugin lives in its own subdirectory under `plugins/sources/` and consists of two
files: a Python implementation (`plugin.py`) and a metadata file (`source.json`).
The plugin loader (`job_sources/loader.py`) scans this directory, validates each
plugin, and registers it with the ingestion pipeline — no changes to core code are
required.

---

## Quick Start

1. **Copy the template folder** to a new directory named after your source:

   ```powershell
   Copy-Item -Recurse plugins\sources\_template plugins\sources\<your_source>
   ```

   The folder name must match the `source_key` you will set in `source.json`.

2. **Rename the class** inside `plugin.py` to something descriptive (e.g.
   `MySourceClient`). There must be exactly one `JobSource` subclass in the file.

3. **Implement the three required methods** in `plugin.py`:

   - `fetch_page(page)` — fetch one page of raw listings from the source API.
   - `total_pages()` — return the total number of pages available.
   - `normalise(raw)` — convert one raw listing dict to the canonical schema.

   See the [Canonical Listing Schema](#canonical-listing-schema) section for the
   exact keys `normalise()` must return.

4. **Fill in `source.json`** with metadata about your source:

   ```json
   {
     "source_key": "your_source",
     "display_name": "Your Source",
     "description": "One-sentence description shown in the Settings UI.",
     "home_url": "https://your-source.example.com",
     "fields": []
   }
   ```

   Use `"fields": []` if the API is public (no credentials needed). Add field
   objects for each credential the source requires. See the
   [`source.json` Reference](#sourcejson-reference) section for details.

5. **Restart `ingest.py` or `app.py`** — the source auto-registers on startup.
   `ensure_plugins_registered()` in `job_sources/auto_register.py` reads the
   plugin registry and writes any new source entries into `config/providers.json`
   with `enabled: false`.

6. **Enable the source** (keyed sources only): navigate to `/settings`, enter the
   required credentials, and toggle the source on. Keyless (public API) sources
   are enabled automatically.

## `source.json` Reference

| Field          | Type             | Description |
|----------------|------------------|-------------|
| `source_key`   | string           | Unique identifier. **Must match the plugin folder name.** Used as the registry key and stored on every listing. |
| `display_name` | string           | Human-readable name shown in the Settings UI. |
| `description`  | string           | One-sentence description shown below the source name in Settings. |
| `home_url`     | string           | URL to the source's homepage. Shown as a link in Settings. |
| `fields`       | array of objects | Credential fields required by this source. Empty array for keyless sources. |

### `fields` entry structure

| Key        | Type    | Description |
|------------|---------|-------------|
| `name`     | string  | Python key used to retrieve the value from the `credentials` dict in `__init__`. |
| `label`    | string  | Human-readable label shown next to the input in the Settings UI. |
| `type`     | string  | `"text"` for plain text; `"password"` for masked input. |
| `required` | boolean | Whether the field must be filled before the source can be enabled. |

The loader reads only the five top-level keys listed above. Any additional keys
are silently ignored during validation but will be attached to `cls._plugin_schema`.

`settings_schema()` is **not** defined in `plugin.py`. It is auto-generated by the
loader from `source.json` and injected onto the class at load time.

## Canonical Listing Schema

`normalise()` must return a dict containing **all** of the following keys. Use
`None` for fields the source does not provide — never omit a key.

| Key            | Type          | Description |
|----------------|---------------|-------------|
| `source`       | str           | Source identifier; must match `source_key`. |
| `source_id`    | str           | Unique listing ID from the source. |
| `title`        | str           | Job title. |
| `company`      | str           | Employer name. |
| `location`     | str           | Location string as returned by the source. |
| `salary_min`   | float or None | Lower salary bound in local currency. |
| `salary_max`   | float or None | Upper salary bound in local currency. |
| `salary_period`| str or None   | Pay period: `"annual"`, `"daily"`, `"hourly"`, or `None`. |
| `contract_type`| str or None   | e.g. `"permanent"`, `"contract"`, or `None`. |
| `contract_time`| str or None   | e.g. `"full_time"`, `"part_time"`, or `None`. |
| `description`  | str or None   | Short snippet or `None`. The pipeline scrapes the full job description later unless `skip_scrape` is `True`. |
| `redirect_url` | str           | URL linking to the full job listing. |
| `created_at`   | str or None   | ISO 8601 string, e.g. `"2026-01-02T12:34:56Z"`. |
| `skip_scrape`  | bool          | Optional (default `False`). Set `True` when the source URL is known to block scrapers (e.g. returns 403). The pipeline uses the API description directly. |
| `description_is_full` | bool   | Optional (default `False`). Set `True` alongside `skip_scrape` when the API provides complete job descriptions (not just snippets). Listings with this flag and descriptions >= 100 chars are classified as `"full"` in the main feed instead of `"snippet"`. |

## Keyless vs. Keyed Sources

**Keyless sources** (`"fields": []`) connect to a public API that requires no
authentication. The loader auto-enables them in `providers.json` — no user action
is needed after restart.

**Keyed sources** (one or more entries in `"fields"`) require credentials (an API
key, username/password, etc.). The loader adds them to `providers.json` with
`"enabled": false`. The user must go to `/settings`, enter the credentials, and
enable the source before it will be used by the ingestion pipeline.

When to use each:

- Use keyless when the API is genuinely public and requires no sign-up.
- Use keyed whenever any credential — even a free-tier API key — is required.
  This ensures credentials are stored in `config/providers.json` (gitignored) and
  never hard-coded.

## Tips

### Caching `total_pages()`

If your source requires a network request to determine the page count (e.g. a
first-page response includes a `total_results` field), cache the result to avoid
redundant requests:

```python
def __init__(self, config, credentials=None):
    ...
    self._total_pages: int | None = None

def total_pages(self) -> int:
    if self._total_pages is None:
        first_page = self._fetch_raw_page(1)
        total = first_page.get("total_results", 0)
        per_page = self._config.get("results_per_page", 10)
        self._total_pages = max(1, -(-total // per_page))  # ceiling division
    return self._total_pages
```

### Using `job_sources/utils.py` helpers

Two helpers in `job_sources/utils.py` cover the most common normalisation tasks:

- `strip_html(html: str) -> str` — strips HTML tags from a description string
  using BeautifulSoup. Use this whenever the source returns HTML in its
  description field.

- `parse_salary(raw: str) -> tuple[float | None, float | None]` — parses a
  free-text salary string (e.g. `"$80,000 - $120,000"`, `"100k"`) into a
  `(salary_min, salary_max)` tuple. Returns `(None, None)` for empty or
  unparseable strings.

```python
from job_sources.utils import strip_html, parse_salary

salary_min, salary_max = parse_salary(raw.get("salary", ""))
description = strip_html(raw.get("description", ""))
```

### Handling pagination

The default `pages()` iterator in `JobSource` calls `total_pages()` once, then
calls `fetch_page(page)` for each page from 1 to `total_pages()`, stopping early
if a page returns an empty list. Most sources can rely on this default.

Override `pages()` only if your source uses 0-based page numbering, requires a
different stopping condition, or needs to cache cross-page state.

### Error handling in `fetch_page()`

Always return an empty list on error — never raise from `fetch_page()`. The
pipeline treats an exception as an unrecoverable failure; returning `[]` allows
it to log a warning and continue with other sources.

```python
def fetch_page(self, page: int) -> list[dict]:
    try:
        resp = requests.get(self._url, params={...}, timeout=15)
    except requests.RequestException as exc:
        logger.warning("MySource request failed: %s", exc)
        return []
    if resp.status_code != 200:
        logger.warning("MySource returned HTTP %d", resp.status_code)
        return []
    return resp.json().get("results", [])
```

## Testing

Validate your `source.json` by running the existing schema test suite:

```powershell
pytest tests/test_source_json.py -v
```

This checks that all required keys are present, the `source_key` matches the
folder name, and every field entry is well-formed.

Write a unit test for your plugin following the pattern of existing source tests
(e.g. `tests/test_job_sources_jobicy.py`). At minimum, cover:

- `normalise()` with a representative raw dict — all canonical keys present, correct types.
- `fetch_page()` error handling — verify it returns `[]` on a network error or non-200 response.
- `total_pages()` — verify it returns the expected integer for a known response fixture.

Name your test file `tests/test_job_sources_<source_key>.py` and use
`unittest.mock.patch` to avoid real network calls.
